# Доступ Package Center через внешний HTTP-прокси

> [!info] Кратко
> На сетях с DPI-фильтрацией (ТСПУ и аналоги) трафик NAS к Cloudflare режется по содержимому, в результате Package Center и `synopkg` зависают.
> **Лечение:** взять готовый HTTP-прокси у VPN-провайдера и прописать его в **DSM → Control Panel → Network → Connectivity → Proxy**. DSM сама направит весь исходящий трафик пакетного слоя через прокси.
> Настройка целиком выполняется в GUI DSM, без работы в терминале.
> **Решение принято в:** [[ADR-0003-package-center-via-external-proxy.ru]].

## Когда применять

✅ Подходит, если:
- На сети между NAS и интернет есть **DPI** (типичный признак: ответы Cloudflare-ресурсов режутся по содержимому, обнаружено в ходе [[ADR-0001-package-center-hangs-on-sectigo-ocsp.ru]]).
- Симптом: Package Center виснет/«Сбой подключения».
- Есть подписка на коммерческий VPN-сервис с режимом HTTP/HTTPS-прокси, либо аналогичный готовый прокси вне зоны DPI.

❌ Не подходит / избыточно, если:
- Сеть без DPI и проблема в чём-то другом (см. [[Solving-fix-stale-update-urls-in-synoinfo.ru]] — может быть достаточно правки URL).

## Что покрывает прокси

| Что | Через прокси? |
|---|---|
| Package Center (GUI) | ✅ |
| DSM Update, Synology Account, push-уведомления и т.п. | ✅ — по принципу «системный HTTP-прокси DSM применяется ко всем исходящим соединениям DSM-сервисов» |
| Файловые шары (SMB/AFP/NFS) клиентам в LAN | ❌ — напрямую |
| Веб-интерфейс DSM клиентам в LAN | ❌ — напрямую |
| Hyper Backup в облако | зависит от назначения и наличия proxy-поддержки в задаче |
| Docker-контейнеры (если работают) | ❌ — в каждом контейнере отдельно настраивать |

То есть локальная сеть и фактическое использование NAS клиентами — не страдает. Прокси — только для **исходящего** трафика DSM-сервисов.

## Шаг 1. Получить параметры HTTP-прокси у VPN-провайдера

Большинство платных VPN-сервисов помимо обычного VPN-тоннеля **раздают HTTP/HTTPS-прокси**. Это отдельный режим, обычно описан в личном кабинете как «HTTP/SOCKS proxy», «Прокси-серверы», «Browser proxy» и т.п. Подходит любой такой сервис, у которого:

- есть **HTTP CONNECT-прокси** (не только SOCKS — DSM SOCKS не поддерживает в этой настройке);
- сервер находится **вне зоны DPI** (страна с открытым интернетом);
- есть **авторизация по логину/паролю** (или whitelisting по IP).

Из личного кабинета VPN-провайдера понадобятся:
- адрес прокси-сервера: `host` или `IP`;
- порт (часто `8080`, `3128`, `8888`, `1080`);
- логин и пароль.

> [!note] Альтернатива — собственный прокси на VPS
> Вместо готового прокси от VPN-провайдера можно поднять собственный HTTP-прокси (`tinyproxy`, `3proxy`, `squid`) на дешёвом VPS вне зоны DPI. Документация — на сайтах этих проектов; в данной инструкции не приводится. Параметры (хост / порт / логин / пароль) для шага 2 в обоих случаях одинаковы.

## Шаг 2. Прописать прокси в DSM

DSM в браузере → **Control Panel → Network → Connectivity → Proxy** (вкладка):

1. Поставить галку **Use proxy server**.
2. **Address:** хост или IP прокси (без `http://`).
3. **Port:** порт прокси (из личного кабинета VPN-провайдера).
4. **Use the same proxy server for all protocols:** ✅ галка.
5. **Authentication required:** ✅ галка → ввести логин/пароль.
6. **Bypass proxy server for local addresses:** ✅ (чтобы LAN не шла через прокси).
7. **Apply** / **OK**.

DSM подхватит настройки сразу, без рестарта.

## Шаг 3. Откатить артефакты от предыдущего решения

Если ранее применялся blackhole/Cloudflare-фикс из [[Solving-package-center-ocsp-cloudflare-blackhole.ru]] — выполните чек-лист отката из раздела [[Solving-package-center-ocsp-cloudflare-blackhole.ru#Откат]]: удалите соответствующие задачи в **Control Panel → Task Scheduler**, верните MTU на Auto, восстановите системные файлы из бэкапов.

Иначе остатки blackhole-маршрутов и заглушек могут конфликтовать с прокси и вызывать симптомы похожие на исходную проблему.

Если предыдущее решение не применялось — этот шаг можно пропустить.

## Шаг 4. Проверка через GUI Центра пакетов

В браузере → закрыть вкладку DSM целиком → открыть DSM → **Package Center**.

Признаки корректной работы:
- Окно открывается **без** баннера «Сбой подключения».
- Вкладка **Все пакеты / All Packages** прогружает каталог Synology.
- Вкладка **Установлено / Installed** показывает корректное состояние пакетов (значки апдейтов, если они есть).
- **Settings (шестерёнка) → Auto Updates → Check Now** возвращает результат без ошибки.

## Что сделать с MTU и DNS, если их меняли

Если в ходе предыдущих экспериментов меняли:

- **MTU eth0** — DSM → Control Panel → Network → Network Interface → LAN 1 → Edit → **MTU value** = `Auto` (или явно 1500).
- **DNS** — DSM → Control Panel → Network → General → **Manually configure DNS server** → вернуть на роутер `192.168.1.1`. Публичный DNS (`1.1.1.1`+`8.8.8.8`) тоже подойдёт.

## Откат

Чтобы отключить прокси и вернуть DSM к прямым соединениям:

DSM → **Control Panel → Network → Connectivity → Proxy** → **снять галку** Use proxy server → **Apply**.

Этого достаточно. Никаких файлов на NAS править не нужно.

## Что делать, если прокси перестал работать

Симптом: Package Center снова виснет или отдаёт «Сбой подключения».

В первую очередь — **через GUI DSM**:
1. **Control Panel → Network → Connectivity → Proxy** — убедиться, что настройка прокси сохранилась (галка **Use proxy server** активна, адрес и порт совпадают с теми, что выдаёт VPN-провайдер).
2. **Личный кабинет VPN-провайдера** — проверить статус подписки, актуальность адреса прокси и доступность серверов. Если провайдер сменил адрес — обновить в DSM.
3. Если у VPN-провайдера несколько серверов — попробовать другой (другая страна/город), обновить адрес в DSM. Это особенно актуально, если прежний IP попал под фильтрацию DPI.

После любого изменения настроек в DSM — переоткрыть Package Center в браузере (закрыть вкладку и открыть заново) и повторить проверку из Шага 4.

## Технические детали

### Почему прокси работает там, где blackhole не помог

Принцип: настройка прокси в DSM применяется **на уровне приложения** — DSM-сервисы (включая Package Center и пакетный демон) при исходящих HTTP/HTTPS-запросах сначала устанавливают TLS-туннель к прокси-серверу, а уже через него ходят на серверы Synology / Cloudflare.

- DPI у провайдера видит только зашифрованный туннель `<NAS> → <прокси>:<порт>`.
- SNI к Cloudflare и сам обмен с серверами Synology происходит **из прокси-сервера**, который находится вне зоны DPI.
- Фильтрация по содержимому DPI не помогает — содержимое внутри HTTPS-туннеля недоступно для анализа.

Это принципиально отличается от попыток локально подменить маршруты или /etc/hosts (см. [[ADR-0001-package-center-hangs-on-sectigo-ocsp.ru]] — там разобрано, почему такие подходы недостаточны против DPI на содержимом TLS-потока).
