# ADR-0003 — Доступ Package Center через внешний HTTP-прокси

> [!summary]
> Корень проблемы зависающего Package Center на DS218play — **DPI-фильтрация трафика к Cloudflare** на стороне провайдера (предположительно ТСПУ). Synology перенесла часть инфраструктуры (`pkgautoupdate.synologyupdate.com`) на Cloudflare без AWS-fallback. Попытки локального обхода (blackhole-маршруты, /32 исключения, MTU) дают только частичный эффект — DPI режет TLS-поток по содержимому.
> Принятое решение: направить исходящий трафик DSM-сервисов через готовый **HTTP-прокси** вне зоны DPI (предоставляется коммерческим VPN-сервисом). Указывается в **DSM → Control Panel → Network → Connectivity → Proxy**.

## Контекст

### Ограничения предыдущего подхода
[[ADR-0001-package-center-hangs-on-sectigo-ocsp]] предполагало, что зависание `synopkg`/Package Center вызывается OCSP-проверкой сертификата `update7.synology.com` — Sectigo-OCSP сидит на Cloudflare, провайдер режет Cloudflare, TLS ждёт OCSP-таймаут. Решение через `ip route blackhole` устраняло симптом OCSP, но:

1. Не покрывало `pkgautoupdate.synologyupdate.com` — этот домен **легитимно нужен** GUI Package Center (302 redirect от `pkgupdate7.synology.com`), и сидит **полностью на Cloudflare**.
2. Узкие /32-исключения для конкретных IP не выдерживают ротации Cloudflare.
3. Главное — DPI режет TLS-поток **по содержимому**, не по адресу: короткие HEAD-запросы к Cloudflare-ресурсам проходят, длинные ответы (более ~16 КБ) обрезаются. Снижение MTU при этом не помогает.

Маршрутизационные обходы не работают против фильтрации по содержимому.

### Что нужно решить
- `synopkg` (CLI) и DSM Package Center (GUI) должны иметь доступ к серверам Synology, включая Cloudflare-ресурсы (`pkgautoupdate.synologyupdate.com` и OCSP/CRL Sectigo).
- Решение должно быть устойчивым к ротации IP, ребуту, обновлениям DSM.
- Не должно требовать постоянных правок iptables / маршрутов / `/etc/hosts`.

### Условия задачи
- На NAS нет нативной поддержки WireGuard, Shadowsocks, Tor.
- Docker на DS218play не работает (RTD1296 неподдерживаемая платформа).
- DSM **умеет** из коробки HTTP CONNECT-прокси для всего исходящего трафика собственных сервисов (Package Center, DSM Update, Synology Account, push) — это встроенная настройка, не требует ничего ставить.
- Доступен готовый HTTP-прокси у коммерческого VPN-сервиса (большинство платных VPN раздают такой режим в личном кабинете). Альтернативно — собственный прокси на VPS вне зоны DPI.

## Рассмотренные варианты

### Вариант A — `ip route blackhole` + /32 для Cloudflare-IP
- **Плюсы:** локально, не требует внешней инфраструктуры.
- **Минусы:** не покрывает `pkgautoupdate.synologyupdate.com`, ротация IP, не справляется с DPI на содержимом.
- **Решение:** **superseded**, см. [[ADR-0001-package-center-hangs-on-sectigo-ocsp]].

### Вариант B — VPN-клиент на NAS (OpenVPN/L2TP)
- **Плюсы:** DSM поддерживает из коробки (Network Interface → Create VPN profile).
- **Минусы:**
  - Если поднять как **default route через VPN** — **весь** трафик NAS пойдёт через VPN, что замедлит локальные операции (бэкапы, медиасервер, файловые шары) и съест трафик VPS.
  - Если поднять с **частичной маршрутизацией** (только Cloudflare/Sectigo через VPN) — DSM GUI этого не позволяет, нужно править таблицы маршрутизации руками после каждого подключения VPN.
  - WireGuard в DSM 7.3 нативно нет — только OpenVPN/L2TP, оба тяжелее.
  - На DS218play (1 ГБ RAM, ARMv8) держать постоянный VPN-туннель — лишняя нагрузка.
- **Решение:** отвергнут как избыточный.

### Вариант C — HTTP CONNECT-прокси, прописанный в настройках DSM ✅
- **Плюсы:**
  - **Из коробки в DSM:** Control Panel → Network → Connectivity → Proxy. Никаких скриптов, маршрутов, файлов.
  - Распространяется только на исходящий трафик DSM-сервисов (Package Center, DSM Update, Synology Account, Active Insight). Локальные шары/бэкапы/медиасервер идут напрямую.
  - В качестве источника прокси может выступать готовый сервис — HTTP-прокси-режим, входящий в подписку коммерческого VPN. Собственную инфраструктуру разворачивать не требуется.
  - Безразлично к Cloudflare-ротации IP, OCSP-инфраструктуре, изменениям инфраструктуры Synology.
  - Не зависит от ребутов NAS — настройка прокси в DSM сохраняется как обычная сетевая настройка.
- **Минусы:**
  - Зависимость от доступности прокси-сервиса и срока подписки на VPN.
  - Если IP прокси попадёт под фильтрацию — потребуется выбрать другой сервер (у VPN-сервисов обычно пул адресов).
- **Альтернатива в рамках того же варианта:** поднять собственный HTTP-прокси (`tinyproxy`, `3proxy`, `squid`) на дешёвом VPS вне зоны DPI. Параметры (хост/порт/логин/пароль) для DSM при этом аналогичны.
- **Решение:** **принят** в реализации с готовым прокси от VPN-сервиса.

### Вариант D — Cloudflare WARP в Docker
- **Плюсы:** бесплатно.
- **Минусы:** Docker на DS218play не работает; WARP при активном DPI часто тоже режется; на 1 ГБ RAM держать второй сетевой стек — тяжело.
- **Решение:** отвергнут.

### Вариант E — обфусцированный VPN (AmneziaWG, XRay/XTLS-Reality)
- **Плюсы:** проектируется под обход DPI.
- **Минусы:** на NAS только через Docker / Entware-сборку, поддерживать сложно, требует больше ресурсов чем простой HTTP-прокси.
- **Решение:** отложен. Если прокси-вариант перестанет работать (DPI начнёт резать сами VPS-IP) — рассмотреть.

## Принятое решение

Применить вариант **C** в реализации с **готовым HTTP-прокси, входящим в подписку коммерческого VPN-сервиса**.

Альтернативная реализация (собственный HTTP-прокси на VPS, например `tinyproxy`) — равноценный путь по сути, шаги настройки в DSM идентичны.

Шаги:
1. Получить параметры HTTP-прокси у VPN-провайдера (адрес, порт, логин, пароль).
2. Прописать прокси в **DSM → Control Panel → Network → Connectivity → Proxy** с авторизацией.
3. Откатить артефакты ADR-0001 (blackhole-маршруты, /32, MTU, Task Scheduler-задачи) — см. [[Solving-package-center-ocsp-cloudflare-blackhole#Откат]].

Подробная инструкция — [[Solving-package-center-external-proxy]].

## Последствия

### Положительные
- Package Center в GUI получает доступ к серверам Synology (включая Cloudflare-ресурсы) через прокси-туннель.
- Решение не зависит от Synology-инфраструктуры (Cloudflare/AWS, OCSP-серверы, ротация IP).
- Не требует поддержки скриптов, Task Scheduler-задач, периодических обновлений списков IP.
- По задумке настройка системного прокси DSM применяется ко всем исходящим соединениям DSM-сервисов (DSM Update, Synology Account, Active Insight, push-уведомления).

### Нейтральные
- Постоянный платёж за подписку на VPN-сервис (обычно уже оплачен для других нужд).
- В **Control Panel → Network → Connectivity → Proxy** видна постоянная настройка с адресом прокси.

### Отрицательные / риски
- **Зависимость от внешнего прокси:** если VPN-сервис упадёт или прекратит услугу — Package Center снова сломается. Митигация: иметь резервный прокси (второй VPN-сервис).
- **Зависимость от того, что DPI не блокирует сам IP прокси.** Если IP попадёт в блок-листы — потребуется сменить адрес. У VPN-провайдеров обычно есть пул адресов.
- **Прокси видит весь HTTPS-трафик DSM** (как зашифрованные пакеты — расшифровать без MitM не может). Тем не менее провайдер прокси технически может логировать destination-домены через SNI. Допустимый риск для домашнего NAS, передающего публичные API-запросы Synology.
- **Прокси не покрывает Docker-контейнеры** (на DS218play это не актуально — Docker не работает) и сторонние процессы, которые читают proxy-настройки **не из** DSM-конфига. Например, прямые HTTPS-вызовы из Python-скриптов в задачах Task Scheduler — для них прокси нужно прописывать отдельно.

### Не покрывается этим решением
- Доступ к торрент-трекерам / соцсетям / других заблокированных ресурсов: эта настройка только для DSM. Для остального — нужен VPN на роутере / отдельные клиенты.
- Доступ из приложений, которые игнорируют системный прокси (некоторые мобильные приложения Synology типа DS file могут ходить напрямую).

## Признаки корректной работы

После применения прокси:
- Package Center в GUI открывается без баннера «Сбой подключения».
- Вкладки «Все пакеты» и «Установлено» прогружают каталог и состояние пакетов.
- **Settings → Auto Updates → Check Now** отрабатывает без ошибок.

Симптомы, которые встречаются при проблеме (для сравнения «до/после»):
- В GUI Package Center — баннер «Сбой подключения», список пакетов не загружается.
- В CLI — `synopkg checkupdateall` уходит в таймаут (`exit=124`, ~30 сек ожидания).
